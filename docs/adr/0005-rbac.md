# ADR 0005: Role-Based Access Control (RBAC)

## Status

Accepted

## Context

MatchHire has three distinct user types with different permissions and access patterns:

- **Candidates**: Can upload resumes, view jobs, apply to jobs, manage their applications
- **Recruiters**: Can post jobs, view applications, schedule interviews, view analytics
- **Admins**: Can manage users, moderate content, view system-wide analytics

We needed a flexible, maintainable authorization system that could:

- Enforce role-based permissions at the view and object level
- Scale to add new roles in the future
- Integrate with Django's permission system
- Be testable and maintainable
- Support both API and admin access

## Decision

Implement Role-Based Access Control (RBAC) using a custom role field on the User model with Django's permission system for fine-grained access control.

### Implementation Details

- **Roles**: CANDIDATE, RECRUITER, ADMIN (defined in User model)
- **Permission Storage**: Django's built-in permission system
- **Permission Enforcement**: Custom permission classes and decorators
- **RBAC Module**: `apps/users/rbac.py` for role-based logic

### Role Definitions

```python
class User(AbstractUser):
    class Roles(models.TextChoices):
        CANDIDATE = "CANDIDATE", "Candidate"
        RECRUITER = "RECRUITER", "Recruiter"
        ADMIN = "ADMIN", "Admin"
    
    role = models.CharField(max_length=20, choices=Roles.choices, default=Roles.CANDIDATE)
```

### Permission Enforcement

```python
# Custom permission classes
class IsRecruiter(permissions.BasePermission):
    def has_permission(self, request, view):
        return request.user.role == User.Roles.RECRUITER

class IsCandidateOrRecruiter(permissions.BasePermission):
    def has_permission(self, request, view):
        return request.user.role in [User.Roles.CANDIDATE, User.Roles.RECRUITER]

# View usage
class JobCreateView(generics.CreateAPIView):
    permission_classes = [IsAuthenticated, IsRecruiter]
```

### Object-Level Permissions

For object-level access control (e.g., recruiters can only edit their own jobs):

```python
class IsJobOwner(permissions.BasePermission):
    def has_object_permission(self, request, view, obj):
        return obj.recruiter == request.user
```

## Alternatives Considered

### Alternative 1: Django Groups and Permissions Only

- **Pros**: Django built-in, standard approach
- **Cons**: More complex setup, harder to reason about, requires admin configuration

### Alternative 2: Third-Party Libraries (django-guardian, django-rules)

- **Pros**: Feature-rich, object-level permissions out of the box
- **Cons**: Additional dependencies, learning curve, may be overkill for our needs

### Alternative 3: Permission-Based (No Roles)

- **Pros**: More granular control
- **Cons**: Complex permission management, harder to understand user capabilities

### Alternative 4: Attribute-Based Access Control (ABAC)

- **Pros**: Most flexible, policy-based
- **Cons**: Complex implementation, overkill for current requirements

## Pros

- **Simplicity**: Easy to understand and reason about
- **Django Integration**: Works with Django's permission system
- **Scalability**: Easy to add new roles by extending the Roles enum
- **Testability**: Straightforward to test role-based access
- **Performance**: Simple field checks, fast permission evaluation
- **Maintainability**: Centralized RBAC logic in one module
- **Flexibility**: Can combine with Django permissions for fine-grained control

## Cons

- **Limited Granularity**: Roles may be too coarse-grained for complex scenarios
- **Role Explosion**: May need many roles as requirements grow
- **Static Roles**: Roles are assigned, not dynamic based on context
- **No Permission Inheritance**: Can't inherit permissions from parent roles

## Future Implications

- May need to add team/organization-based permissions for enterprise features
- Consider adding permission caching for high-traffic endpoints
- May need to implement permission inheritance for role hierarchies
- Consider adding audit logging for permission changes
- May need to support multiple roles per user (e.g., user can be both candidate and recruiter)
