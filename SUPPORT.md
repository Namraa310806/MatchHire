## Community Support

### GitHub Issues

For bug reports, feature requests, and questions:

1. Check existing [GitHub Issues](https://github.com/your-org/matchhire/issues) to see if your question has already been addressed
2. If not, create a new issue with:
   - A descriptive title
   - Clear description of the problem or question
   - Steps to reproduce (for bugs)
   - Environment information (Django version, Python version, OS)
   - Relevant code snippets or error messages

### GitHub Discussions

For general questions, ideas, and discussions:

- Use [GitHub Discussions](https://github.com/your-org/matchhire/discussions) for:
  - "How do I..." questions
  - Architecture discussions
  - Feature ideas
  - Community announcements

### Stack Overflow

Tag your questions with `matchhire` and `django` for visibility.

## Bug Reports

When reporting bugs, please include:

### Required Information

- MatchHire version
- Django version
- Python version
- Operating system
- Steps to reproduce the issue
- Expected behavior
- Actual behavior
- Error messages or stack traces

### Optional but Helpful

- Minimal code example to reproduce the issue
- Screenshots (if applicable)
- Configuration files (with secrets redacted)
- Logs (with sensitive information redacted)

### Bug Report Template

```markdown
## Description
Brief description of the bug

## Steps to Reproduce
1. Step one
2. Step two
3. Step three

## Expected Behavior
What should happen

## Actual Behavior
What actually happens

## Environment
- MatchHire version: x.x.x
- Django version: x.x.x
- Python version: x.x.x
- OS: [OS name and version]

## Additional Context
Any other relevant information
```

## Feature Requests

We welcome feature requests! When requesting a feature:

1. Check the [ROADMAP](ROADMAP.md) to see if it's already planned
2. Search existing issues to avoid duplicates
3. Provide a clear use case for the feature
4. Consider if you're willing to contribute the feature

### Feature Request Template

```markdown
## Feature Description
Clear description of the feature

## Use Case
Why is this feature needed? What problem does it solve?

## Proposed Solution
How do you envision this feature working?

## Alternatives Considered
What alternatives did you consider? Why did you reject them?

## Additional Context
Any other relevant information, mockups, examples, etc.
```

## Contributing

## Reporting Security Issues

**Do not** report security vulnerabilities via public issues. See [SECURITY.md](SECURITY.md) for the responsible disclosure process.

## Response Time

We aim to respond to:

- **Security issues**: Within 48 hours
- **Bug reports**: Within 7 days
- **Feature requests**: Within 14 days
- **General questions**: Within 7 days

Response times may vary based on maintainers' availability.

## Documentation Issues

If you find errors or omissions in the documentation:

1. Check if the documentation is in the repository
2. If yes, submit a pull request with the fix
3. If no, open an issue describing the documentation problem
